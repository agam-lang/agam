# Phase 17F: Standard Library and Native I/O Expansion

## Status

- Overall status: partial
- Current note: `agam_std::io` now provides the first deterministic file/path I/O slice and `agam_sema::effects` now exposes a matching `FileSystem` effect definition, but runtime-backed effect handling, networking, and broader standard-library packaging work are still open.

## Goal

- Expand `agam_std` and native I/O capability in a way that fits the new package ecosystem and Agam's effects model.
- Grow standard-library depth without importing a messy async/runtime story from other ecosystems.

## Main Motive

- A strong package ecosystem still needs a credible first-party standard library.
- Networking, file I/O, and higher-level batteries should feel language-native, not like thin wrappers around foreign runtime models.

## Responsible Crates

- `agam_std`
- `agam_pkg`
- `agam_sema`

## Planned Work

- Expand `agam_std` so networking, file I/O, and related workflow capability are handled natively through algebraic effects.
- Align standard-library packaging and versioning with the new manifest, lockfile, registry, and environment contracts.
- Keep the standard library modular enough that first-party base distributions can compose it cleanly.

## Checklist Status

- [x] Add an initial first-party file/path I/O slice in `agam_std`.
- [x] Keep the first slice deterministic and small enough to compose under later effect handlers.
- [x] Add a first reusable sema-side filesystem effect definition that matches the shipped stdlib I/O surface.
- [ ] Route native I/O and networking through the effects model instead of only direct Rust helpers.
- [ ] Align broader standard-library packaging/versioning with the new package-governance and distribution contracts.

## Implemented Slices

- Added `agam_std::io` as the first native I/O module with deterministic path inspection helpers plus UTF-8 text-file operations: `exists`, `is_file`, `is_dir`, `create_dir_all`, `read_to_string`, `read_lines`, `write_string`, `append_string`, and `list_dir`.
- Added a structured `IoError` that preserves the failing operation, path, and message instead of collapsing filesystem failures into opaque strings.
- Added crate-level tests for text round trips, append ordering, deterministic directory listing, and missing-file diagnostics so this first slice is covered before the later effects-facing surface exists.
- Added `std_filesystem_effect()` plus `EffectRegistry::register_std_effects()` in `agam_sema`, so the semantic layer now has a first reusable filesystem effect contract that lines up with the current `agam_std::io` operations.

## Remaining Work

- The current stdlib module and sema effect definition now line up, but they are not yet connected to lowering/runtime-backed handler execution.
- Networking, process, and broader workflow capability still need a native Agam-facing design instead of only filesystem helpers.
- Packaging/versioning and first-party distribution composition for the growing standard library still need to be made explicit on top of the 17A-17E package contracts.

## What To Build Next

- Build the first effects-aware file/networking contract on top of this small `agam_std::io` slice instead of bypassing it with ad hoc runtime helpers.
- Keep growing native I/O and networking through the effects model in a way the package manager can distribute reproducibly.
- Keep `agam_std` growth aligned with official package governance rather than one-off ad hoc modules.

## Dependencies And Prerequisites

- package ecosystem phases 17A through 17E
- effects system from Phase 9

## Agent Interpretation

- This phase should grow capability without recreating Python-style runtime sprawl.
- Standard-library expansion should build on the package ecosystem, not bypass it.
