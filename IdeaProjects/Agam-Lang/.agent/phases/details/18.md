# Phase 18: Build an Agam Execution Tool

## Status

- Overall status: partial

## Goal

- Build a secure execution interface so agent frameworks can run Agam code directly.

## Main Motive

- Make Agam usable as a first-class execution language for LLM agents instead of forcing those systems back to Python.

## Responsible Crates

- `agam_driver`
- `agam_runtime`
- `agam_notebook`

## Planned Work

- Create a secure sandboxed tool where an LLM can input a string of Agam code.
- Use the core workflow:
- write the code to a temporary `.agam` file
- invoke `agamc run`
- return `stdout` and `stderr` back to the calling agent system

## Checklist Status

- [x] Add a dedicated agent-facing execution command instead of hiding the contract under `agamc repl --json`.
- [x] Keep the execution request/response contract strict and machine-consumable.
- [x] Support source-from-stdin and source-from-file flows without bypassing the shared headless execution engine.
- [x] Add stronger request-level resource and capability controls beyond the current temp-workspace boundary.
- [ ] Add stronger OS-level isolation beyond the current request-policy contract.

## Implemented Slices

- Added `agamc exec` as the dedicated execution-tool surface on top of the existing `HeadlessExecutionRequest` and `HeadlessExecutionResponse` contract.
- Kept backward compatibility with `agamc repl --json`, but stopped requiring agent workflows to pretend the REPL is the product surface.
- Taught `agamc exec` to accept either a strict JSON request over stdin or direct source via stdin, `--source`, or `--file`, while still returning one structured JSON response with `success`, `exit_code`, `stdout`, `stderr`, and optional `error`.
- Reused the sanitized filename, temporary workspace materialization, and in-process JIT/LLVM/C execution path already built in Phase 16 instead of inventing a second execution engine.
- Added explicit `HeadlessExecutionPolicy` limits for source bytes, arg count, total arg bytes, and native-backend opt-in, and enforced those limits in `agamc exec` / REPL normalization instead of relying only on a temp workspace plus filename sanitization boundary.

## Remaining Work

- The current execution boundary is now policy-shaped and machine-enforced, but it is not yet a full OS-level sandbox. Stronger isolation around filesystem, network, process, and runtime resource usage still needs explicit platform-level enforcement work.

## What To Build Next

- Add OS-level isolation on top of the current request-policy contract.
- Keep extending policy controls only when they are explicitly enforceable by the runtime or driver.
- Reuse this command surface from `agam_ffi` and later ecosystem wrappers instead of making wrappers speak REPL-specific semantics.

## Dependencies And Prerequisites

- REPL/headless execution groundwork from Phase 16
- stable CLI/run behavior

## Agent Interpretation

- This is the first direct bridge from Agam to external agent workflows.
