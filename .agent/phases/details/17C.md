# Phase 17C: Registry Index and Publish Protocol

## Status

- Overall status: completed

## Goal

- Define the central Agam registry-index protocol and the immutable publish contract for source packages.
- Make package identity registry-based instead of coupling it to Git repository layout.

## Main Motive

- Agam needs one discoverable ecosystem index, but it should stay thin, auditable, and easy to mirror.
- Publishing and installation should be shaped around immutable releases and verified metadata instead of ad hoc cloning.

## Responsible Crates

- `agam_pkg`
- `agam_driver`

## Planned Work

- Define a central index repository layout and release-metadata schema.
- Add publish, install, update, inspect, and audit flows around immutable package releases.
- Support official packages inside the Agam org without forcing every package to live in a separate repository.
- Keep the registry protocol simple enough for mirrors and private indexes later.

## Implemented Slices

- Added the initial Phase 17C registry contract to `agam_pkg`: `RegistryConfig`, `RegistryPackageEntry`, `RegistryRelease`, `PublishManifest`, `PublishReceipt`, index sharding, package-name validation, config/entry read-write helpers, and immutable release append/publish helpers.
- Upgraded registry dependency resolution so a configured local index can provide real release metadata and transitive dependency references instead of always emitting a placeholder record.
- Added `agamc publish` in `agam_driver` to validate a workspace-rooted `agam.toml`, derive a source-package checksum from workspace sources, apply publish-time metadata overrides, bootstrap a local `config.json` when needed, and either print the publish contract with `--dry-run` or append an immutable release to the index.
- Added `agamc registry inspect` and `agamc registry audit` in `agam_driver` so local registry indexes can now be queried and audited through the first-party CLI instead of only by direct file inspection.
- Added `agamc registry install` and `agamc registry update` in `agam_driver` so workspaces can pin and refresh source-package dependencies directly from a selected local registry index while immediately regenerating `agam.lock` against that same index.
- Extended `RegistryRelease` with optional download URLs plus release provenance (`source_checksum`, `manifest_checksum`, `published_by`, `source_repository`) and added `set_registry_release_yanked()` so the local index can now carry explicit release download metadata and yanking state instead of only a bare checksum.
- Added `agamc publish --download-url` and `agamc registry yank` so local registry flows can record download metadata during publish and mark/unmark immutable versions as yanked without mutating the rest of the release contract.
- Added focused driver and package tests covering local-index resolution, publish metadata persistence, immutable version enforcement, and dry-run non-mutation.

## What To Build Next

- Keep the registry protocol thin and mirrorable while the higher-level package UX grows around it.
- Push remaining package-ecosystem work into Phase 17D environments and later remote/governance follow-ups instead of bloating the local index contract.

## Remaining Work

- The current publish path targets a local filesystem registry index; there is no remote registry API or download flow yet.
- Publish/download metadata is now recorded in the index, but artifact transport is still metadata-only until a real remote registry or hosted mirror path exists.
- Official-package governance, richer provenance attestation, and broader registry operations remain follow-up slices outside the completed local-index contract.

## Dependencies And Prerequisites

- manifest contract from 17A
- resolver and lockfile contract from 17B

## Agent Interpretation

- The registry should be thin and boring by design.
- Avoid tying package identity to repo naming conventions or one-repo-per-package assumptions.
