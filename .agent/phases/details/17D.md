# Phase 17D: Environments and SDK Linking

## Status

- Overall status: open

## Goal

- Build named Agam environments that pin compiler version, SDK packs, target packs, runtime expectations, and dependencies together.
- Make per-project environments reproducible without turning Agam into a clone of Conda.

## Main Motive

- Users need a clean way to switch between host-native, Android, data-science, and release environments without manually wiring toolchains each time.
- Environment state should be explicit project metadata, not hidden shell-global mutation.

## Responsible Crates

- `agam_pkg`
- `agam_driver`
- `agam_runtime`

## Planned Work

- Define named environments such as `dev`, `test`, `release`, and target-specific profiles.
- Bind environments to compiler versions, SDK packs, target triples, runtime backends, and dependency locks.
- Keep environment activation and inspection first-party through Agam tooling.
- Align environment metadata with `agamc doctor`, SDK discovery, and future background prewarm work.

## What To Build Next

- Start with environment data models and explicit selection rules.
- Make SDK and target-pack linkage reproducible before adding shell integration or convenience shortcuts.
- Keep environments layered on top of manifests and lockfiles rather than inventing a second resolver.

## Dependencies And Prerequisites

- manifest contract from 17A
- lockfile contract from 17B
- registry metadata from 17C
- SDK contract from 15H

## Agent Interpretation

- This is an environment-management phase, not a system-package-manager phase.
- Interoperate with foreign toolchains, but do not absorb foreign package managers into Agam's core contract.
