# Phase 17A: Workspace Contract and Dependency Manifests

## Status

- Overall status: partial
- Current note: `agam_pkg` now carries the first source-workspace, dependency, lockfile, and environment data models

## Goal

- Define one stable `agam.toml` contract for package metadata, workspace layout, dependency declarations, and toolchain expectations.
- Separate source-package metadata from portable runtime package artifacts and SDK distribution manifests.

## Main Motive

- Stop each tool from inventing its own package and workspace assumptions.
- Make package identity, dependency declaration, and project layout deterministic before a resolver or registry is built on top.

## Responsible Crates

- `agam_pkg`
- `agam_driver`
- `agam_lsp`
- `agam_fmt`
- `agam_test`

## Planned Work

- Define the first stable source-package manifest schema for `agam.toml`.
- Add package identity, version, workspace members, dependency sections, target intent, and toolchain requirement fields.
- Keep the manifest aligned with the existing first-party layout from Phase 15G instead of creating a second project model.
- Define diagnostics and validation rules so malformed manifests fail clearly through first-party tooling.

## Checklist Status

- [x] Add the first `agam_pkg` data models for source manifests, dependency specs, lockfiles, and environments.
- [ ] Wire the manifest contract into `agamc new`, workspace discovery, and first-party diagnostics.
- [ ] Freeze the manifest naming and compatibility policy before building resolver flows on top.

## Implemented Slices

- Added `WorkspaceManifest`, `DependencySpec`, `ToolchainRequirement`, `EnvironmentSpec`, and `WorkspaceLockfile` in `agam_pkg`.
- Added TOML read/write helpers plus round-trip tests for manifest and lockfile compatibility, including the current `[project]` manifest shape.

## Remaining Work

- Reuse the new manifest contract from `agam_driver` instead of hand-rendering `agam.toml`.
- Define validation and user-facing diagnostics around malformed dependency and environment metadata.

## What To Build Next

- Freeze the manifest shape and naming conventions first.
- Cover local path dependencies and workspace members before solving remote resolution.
- Reuse the same manifest contract across CLI, formatter, LSP, test, and future daemon surfaces.
