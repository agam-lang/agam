# Phase 17: Package Ecosystem, Registry, and Reproducible Environments

## Status

- Overall status: open

## Goal

- Build Agam into a coherent package ecosystem with a stable source manifest, deterministic resolution, registry protocol, reproducible environments, and curated first-party distributions.
- Keep portable runtime packages, SDK/toolchain packs, source dependencies, and local environments as separate contracts instead of one overloaded format.

## Main Motive

- Stop package, SDK, workspace, and environment concerns from collapsing into one messy tool story.
- Make Agam reproducible across teams and compute environments without copying Cargo, pip, or Conda blindly.

## Responsible Crates

- `agam_pkg`
- `agam_driver`
- `agam_runtime`
- `agam_lsp`
- `agam_fmt`
- `agam_test`
- `agam_std`

## Planned Work

- Split the package ecosystem into subphases 17A through 17F so manifest, lockfile, registry, environment, official-distribution, and standard-library work can advance without being conflated.
- Preserve the existing Phase 15A portable package/runtime contract and Phase 15H SDK distribution contract instead of reopening them casually.
- Build the future package manager on content-addressable, reproducible metadata rather than ad hoc install scripts or repo-name coupling.

## What To Build Next

- Start with 17A and freeze the workspace/package manifest surface.
- Then land 17B deterministic resolution and lockfile behavior before building registry or environment UX.
- Treat 17C through 17E as product layers built on top of those contracts.
- Continue the standard-library and native I/O growth in 17F after the package layers stop drifting.

## Dependencies And Prerequisites

- stable workspace contract from 15G
- daemon/workspace reuse from 15F
- portable package/runtime contract from 15A
- cache contract from 15B
- SDK/target-pack contract from 15H
- effects system from Phase 9

## Agent Interpretation

- Do not mix source dependencies, portable runtime packages, SDK packs, and environments into one format.
- The highest-leverage near-term package work is contract definition and reproducibility, not wrapper count or installer sprawl.
