# Phase 17E: First-Party Base Distributions and Official Packages

## Status

- Overall status: completed

## Goal

- Ship curated first-party base distributions and define how official Agam packages live inside the organization.
- Make the default Agam install feel coherent without turning the base tool into a giant all-in-one distribution.

## Main Motive

- Users need an obvious starting point, but Agam should not bundle every SDK, package, and foreign integration by default.
- Official package governance has to be clear before the ecosystem grows across many repositories.

## Responsible Crates

- `agam_pkg`
- `agam_driver`
- `agam_std`
- `agam_ffi`

## Planned Work

- Define curated first-party package profiles such as base language support, systems/native interop, and data/AI stacks.
- Define official package naming, release, and repository governance inside the Agam organization.
- Keep foreign-language interop as explicit packs and bridges, not as the base package-manager contract.
- Align first-party distributions with the registry and environment model instead of bypassing them.

## Checklist Status

- [x] Define the first curated first-party profile taxonomy on top of the registry contract.
- [x] Define official package naming, publication, and repository governance for the `agam` registry.
- [x] Add first-party CLI inspection flows for curated profiles and governance.
- [x] Add curated profile installation on top of manifest + lockfile refresh instead of a separate package path.
- [x] Add explicit official-package publish validation for reserved `agam-` packages.

## Implemented Slices

- Added `FirstPartyDistributionProfile`, `FirstPartyPackageRecommendation`, and `OfficialPackageGovernance` in `agam_pkg` with the first curated `base`, `systems`, and `data-ai` profile taxonomy plus read-only governance metadata for the canonical `agam` registry namespace.
- Added official-package validation helpers in `agam_pkg` so reserved `agam-` names can publish only through the canonical registry, an approved owner handle, and an Agam-organization repository path.
- Added `agamc registry governance`, `agamc registry profile list`, and `agamc registry profile inspect` in `agam_driver` so the first-party distribution contract is visible through the CLI instead of only through library APIs.
- Added `agamc registry profile install` in `agam_driver`, reusing the registry-backed manifest + lockfile refresh path to pin the curated profile packages into `agam.toml`.
- Extended `agamc publish` with `--official` so first-party packages can publish through an explicit governance-aware path instead of bypassing the reserved-prefix contract.

## What To Build Next

- Keep the curated first-party story layered on top of the registry and environment model instead of inventing a parallel install mechanism.
- Grow official-package automation and hosted distribution only after the governance contract stays stable in local first-party tooling.
- Keep the default install small and layer larger profiles through explicit environment selection.

## Remaining Work

- Curated profiles currently drive manifest/lockfile installation guidance; they do not yet ship hosted bundles or remote bootstrap automation.
- Official-package governance is enforced in the first-party CLI and package contract, but remote registry authorization and organization-level automation remain future work.

## Dependencies And Prerequisites

- registry contract from 17C
- environments from 17D
- FFI/runtime interoperability groundwork

## Agent Interpretation

- The goal is first-party coherence, not maximum bundle size.
- Curated profiles should reduce ecosystem confusion, not hide real dependency boundaries.
