# Phase 17B: Deterministic Resolver and Lockfile

## Status

- Overall status: complete

## Goal

- Build deterministic dependency resolution and a stable `agam.lock` format for reproducible builds.
- Make content addressing and immutable resolution records part of the default workflow.

## Main Motive

- A package ecosystem is not trustworthy until teams can reproduce the same dependency graph and tool inputs exactly.
- Resolver behavior must be explicit, inspectable, and stable before registry growth scales the problem.

## Responsible Crates

- `agam_pkg`
- `agam_driver`
- `agam_runtime`

## Planned Work

- Define the `agam.lock` format and the resolved-package record shape.
- Support deterministic resolution for workspace, path, git, and registry dependencies.
- Use content hashes and release metadata to make package retrieval verifiable.
- Keep lockfile behavior aligned with package caches and future daemon prewarm work.

## Implemented Slices

- Added `WorkspaceLockfile`, `LockedWorkspace`, `LockedPackage`, and `LockedEnvironment` in `agam_pkg`.
- Added deterministic resolution for workspace, path, git, and registry dependencies, including transitive path-dependency walks through shared session metadata.
- Added `read_lockfile_from_path()`, `write_lockfile_to_path()`, `generate_or_refresh_lockfile()`, and `lockfile_content_drift()` in `agam_pkg`.
- Wired lockfile refresh through `agamc lock` plus the main `agamc build`, `check`, and `dev` workflows so `agam.lock` is part of the default first-party path.
- Tightened lockfile freshness so package aliases, dependency source selectors, and version requirements now invalidate stale lockfiles instead of only comparing added/removed package-name sets.
- Tightened lockfile freshness and human-readable diagnostics so named environment records now invalidate stale lockfiles when explicit compiler, SDK, target, backend, or locked-package selections drift from `agam.toml`.
- Tightened `generate_or_refresh_lockfile()` so live path-dependency content drift now forces an automatic lockfile rewrite instead of only emitting a warning after the fact.
- Kept workspace-member and shared-session metadata on the same freshness contract, so resolver reuse does not quietly drift away from manifest, path-dependency, or environment changes.

## Remaining Work

- No blocking work remains inside the original Phase 17B scope. Future phases should extend the same lockfile/freshness contract when they add new locked metadata.

## What To Build Next

- Keep workspace-member, manifest, environment, and later registry surfaces consuming the same resolver and lockfile contract.
- Extend the lockfile only through explicit later phases that also update freshness diagnostics and compatibility tests.
